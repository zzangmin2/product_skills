# 시나리오 문서 작성 스킬

기능 배경이나 아이디어를 받아서, 개발·디자인팀이 실제로 사용할 수 있는
구체적인 시나리오 문서로 만들어준다.

---

## 입력 형식

아래 항목을 받아서 작성한다. 많을수록 좋지만, 배경과 기능 개요만 있어도 시작할 수 있다.
빠진 항목은 먼저 물어보지 말고, 합리적으로 추론해서 작성한 뒤 미결 사항에 표기한다.

```markdown
## 기능명
[기능 이름]

## 배경 및 목적
[이 기능이 왜 필요한지, 어떤 문제를 푸는지]

## 대상 사용자
[누가 이 기능을 사용하는지]

## 현재 생각하는 단계 구성 (선택)
[이미 생각해둔 단계가 있으면 작성]

## 기타 조건 (선택)
[특별히 반영해야 할 제약, 정책, 보상 등]
```

---

## 출력 형식

```markdown
# [서비스명] — [기능명] 시나리오

## 전체 플로우

[단계1] → [단계2] → ... → 완료

[심사나 대기가 있는 경우: 각 단계 처리 소요 시간 명시]

---

## [N단계]. [단계명]

### [소제목]
[세부 내용 — 필요 항목, 조건, 기준 등]

### UX 포인트
- [이 단계에서 중요한 화면 처리 방식]

---

## 완료

### 완료 화면
- [완료 후 사용자에게 노출되는 내용과 다음 액션]

### 보상 (있는 경우)
- [지급 보상 및 조건]

---

## 전체 예상 소요 시간

| 단계 | 사용자 소요 시간 | 대기 시간 |
|------|----------------|----------|
| ... | ... | ... |
| **전체** | **...** | **...** |

---

## 미결 사항

- [ ] [결정이 필요한 항목] — [결정 필요 주체]
```

## 추가 질문 기준

입력이 모호하면 바로 작성하지 않는다. 시나리오 품질에 직접 영향을 주는 항목이 불분명할 때는 먼저 질문한다.

### 질문이 필요한 상황

아래 중 하나라도 해당하면 작성 전에 질문한다.

- **대상 사용자가 불분명한 경우** — 누가 이 플로우를 거치는지 모르면 단계 설계 자체가 달라진다.
- **핵심 목적이 불분명한 경우** — "가입 시나리오 만들어줘"처럼 이 기능이 무엇을 보장하려는지 알 수 없을 때.
- **단계 간 경계가 모호한 경우** — "검증하고 심사한다"처럼 어디서 끊기는지 불분명할 때.
- **분기 조건이 있을 것 같지만 언급이 없는 경우** — 합격/불합격, 정상/예외 케이스 등 결과에 따라 흐름이 갈리는 기능인데 조건이 없을 때.

### 질문 방식

- 한 번에 최대 3개까지만 질문한다. 많으면 사용자가 부담을 느낀다.
- 질문은 구체적으로 쓴다. "더 알려주세요"가 아니라 "합격/불합격 시 각각 어떻게 처리되나요?" 처럼.
- 질문 앞에 왜 이게 필요한지 한 줄로 이유를 붙인다. 맥락 없는 질문은 사용자가 무엇을 답해야 할지 모른다.

**질문 예시:**
```
시나리오를 구체화하기 전에 몇 가지 확인할게요.

1. 이 플로우의 대상이 누구인가요?
   (단계 구성이 대상에 따라 크게 달라져서요.)

2. 심사 결과가 합격/불합격으로 나뉘나요, 아니면 단순 완료인가요?
   (실패 케이스 처리 방식을 결정하는 데 필요합니다.)

3. 완료 후 사용자에게 즉시 권한을 주나요, 아니면 추가 승인 단계가 있나요?
```

### 질문 없이 추론해도 되는 상황

아래 항목은 질문 없이 합리적으로 추론해서 채우고, 미결 사항에 표기한다.

- 소요 시간 추정
- UX 처리 방식 (모달, 알림, 상태 표시 등)
- 세부 유효성 조건 (파일 크기, 형식 등)
- 보상 금액이나 정책 세부 내용

---

## 작성 기준

### 구체화 원칙

입력이 rough할수록 더 많은 걸 채워야 한다. 추론할 때 아래 기준을 따른다.

- **왜 이 단계가 필요한가** — 각 단계는 존재 이유가 있어야 한다. 서비스가 이 단계를 통해 무엇을 확인하거나 보장하려는지를 반영한다.
- **사용자 입장의 흐름** — 서비스 관점의 절차 나열이 아니라, 사용자가 어떤 순서로 무엇을 경험하는지 중심으로 서술한다.
- **개발·디자인이 바로 쓸 수 있는 수준** — "간단히 보여준다" 같은 모호한 표현 대신, "카메라 촬영 또는 갤러리 선택, PDF/JPG/PNG 지원, 10MB 이하"처럼 구체적으로 쓴다.

### 각 단계에서 채워야 할 내용

- 사용자가 이 단계에서 해야 하는 것 (필수/선택 구분)
- 통과/완료 조건
- 실패·거부 시 처리 방식
- 자동 처리 가능한 부분 vs 수동 처리가 필요한 부분
- 이 단계에서 중요한 UX 처리 (모달, 상태 표시, 알림 등)

### 미결 사항

시나리오를 작성하면서 팀 내 결정이 필요한 항목이 반드시 생긴다.
추론해서 채운 항목 중 불확실한 것, 정책 결정이 필요한 것을 빠짐없이 정리한다.

---

## 주의사항

- 입력이 불완전해도 멈추지 않는다. 합리적으로 추론해서 채우고, 불확실한 부분은 미결 사항에 표기한다.
- 단계 수는 사용자 부담을 고려해 꼭 필요한 것만 포함한다. 비슷한 성격의 단계는 합친다.
- 완료 보상이 있다면 온보딩 투자 비용(시간·노력) 대비 납득 가능한 수준인지 함께 검토한다.
- 소요 시간 추정은 실제 사용자 행동 기준으로 현실적으로 작성한다.
