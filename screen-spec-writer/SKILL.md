# 화면 스펙 문서 작성 스킬

사용자가 제공한 플로우 정보와 현재 화면 설명을 바탕으로
Storyboard와 Description을 작성한다.

---

## 입력 형식

사용자에게 아래 형식으로 정보를 요청한다.
항목이 빠져 있으면 작성 전에 먼저 물어본다.

```markdown
## 기능명
[화면이 속한 기능의 이름]

## 기능 개요
[이 기능이 무엇을 하는지 1~3줄 설명]

## 화면 플로우
1. [화면명]
2. [화면명] ← 현재
3. [화면명]
...

## 현재 화면
[플로우에서 몇 번째 화면인지, 화면 이름]

## 화면 구성 요소
[이 화면에 있는 주요 UI 요소들: 버튼, 리스트, 모달, 입력창 등]

## 인터랙션 및 특이사항
[분기 조건, 예외 처리, 디자인 가이드 관련 특이사항 등]
```

---

## 출력 형식

아래 형식을 그대로 사용한다. 항목 순서와 구조를 바꾸지 않는다.

```markdown
### [화면명]

**STORYBOARD**
[이 화면이 플로우에서 어떤 역할을 하는지 1~2문장으로 서술.
앞 화면에서 어떻게 진입하는지, 이 화면에서 무엇을 하는지, 다음 화면으로 어떻게 이어지는지를 포함한다.]

---

**Description**

1. [UI 요소명 또는 동작명]
   - [세부 동작 또는 조건]
   - [세부 동작 또는 조건]

2. [UI 요소명 또는 동작명]
   - [세부 동작 또는 조건]

...
```

---

## 작성 기준

### Storyboard 작성 기준

- 이 화면이 전체 플로우에서 어떤 위치에 있는지 한눈에 파악되게 쓴다.
- 진입 조건(앞 화면에서 어떤 액션으로 오는지)과 이탈 조건(다음 화면으로 어떻게 넘어가는지)을 포함한다.
- 기능 설명이 아니라 **사용자 관점의 흐름**으로 서술한다.
- 1~2문장, 최대 3문장을 넘지 않는다.

**좋은 예:**
> 과외 신청 메인 화면에서 자료 미리보기를 탭하면 사이드바로 진입합니다. 선생님이 구성한 문제 목록과 과외 조건·목표·숙제를 확인하고, 문제를 선택해 다음 단계로 넘어갑니다.

**나쁜 예:**
> 이 화면은 사이드바 영역입니다. 다양한 정보를 확인할 수 있습니다.

---

### Description 작성 기준

- **번호는 스크린샷에 표시된 번호만 사용한다.** 임의로 번호를 추가하거나 순서를 바꾸지 않는다.
- 항목명은 `[닫기]`, `[저장]`, `[문제 목록]`처럼 UI 요소명을 대괄호로 표기하거나, `터치 시 동작`, `선택 여부에 따른 분기`처럼 동작 단위로 쓴다.
- 세부 내용은 들여쓰기 bullet로 작성한다.
- 분기 조건이 있으면 반드시 포함한다. (ex. 선택 O / 선택 X, 정답 / 오답)
- 다른 화면 참조가 필요한 경우 `<본 문서 pXX>` 형식으로 표기한다.
- 개발자와 디자이너가 읽었을 때 구현 기준이 명확해야 한다.

### 스크린샷 번호 해석 규칙

스크린샷에는 원형 번호(①②③…)와 함께 빨간 선(포인터)이 표시된다. 번호를 해석할 때 아래 규칙을 따른다.

**A 기호 (디자인 가이드)**
- 번호 옆에 A 기호가 함께 붙어 있으면 디자인 상태 변화(컬러, 활성/비활성 등)에 대한 설명이다.
- Description에 포함하지 않는다. 언급 없이 스킵한다.

**하위 번호 (1-1, 1-2, 3-1 등)**
- 상위 번호의 하위 항목으로 묶지 않는다.
- 각각 독립된 번호 항목으로 개별 처리한다.

**같은 번호가 한 페이지에 두 번 이상 등장하는 경우**
- 동일한 번호가 여러 곳에 붙어 있으면 같은 동작·의미를 공유한다는 뜻이다.
- Description 항목 하나로 묶어서 설명한다.

**빨간 선 없이 UI 위에 걸쳐 있는 번호**
- 특정 요소를 가리키는 것이 아니라, 번호가 걸쳐 있는 UI 영역 전체를 설명한다는 뜻이다.
- 해당 영역 전체의 동작을 항목으로 작성한다.

**내용을 확신할 수 없는 경우**
- 스크린샷만으로 동작이나 조건을 파악하기 어려운 항목은 내용을 추측해서 쓰지 않는다.
- `- **확인 필요:** [모르는 내용]` 형식으로 솔직하게 표시한다.

### 표기 규칙

**UI 요소명**
- 화면에 표시된 텍스트 그대로 대괄호로 감싸고 bold 처리한다.
- 뒤에 "버튼", "탭" 등의 단어는 붙이지 않는다.
- 예: **[모두 사용]**, **[결제 및 과외 신청]**, **[닫기]**

**항목 예시:**
```
1. 날짜·시간 영역
   - 선택한 과외 날짜 및 시간 노출

1-2. 과외 과목
   - 선택한 과목 노출

2. **[?]**
   - 터치 시 **확인 필요:** 어떤 안내가 노출되는지

3-1. 티켓 수량 스테퍼
   - 최소 0개, 최대 보유 수량까지 조절

3-2. **[모두 사용]**
   - 터치 시 보유 수량 전체 자동 입력

6. **[결제 및 과외 신청]**  ← 빨간 선 없이 버튼 위에 걸쳐 있는 경우
   - 필수 항목 모두 완료 시 활성화
   - 터치 시 결제 처리 화면으로 이동
```

---

## 주의사항

- 플로우에서 현재 화면의 앞뒤 맥락을 반드시 반영한다. 단독으로 읽히는 설명은 쓰지 않는다.
- 사용자가 "특이사항 없음"이라고 해도, 플로우 상 자연스럽게 발생할 수 있는 예외(뒤로가기, 빈 상태, 로딩 등)는 Description에 포함한다.
- 여러 화면을 한 번에 요청받으면, 화면별로 구분선(`---`)을 두고 순서대로 작성한다.
- 출력 후 사용자에게 "추가하거나 수정할 항목이 있으면 말씀해주세요"로 마무리한다.
