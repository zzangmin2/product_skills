# 시나리오 QA 스킬

서비스 기획 단계에서 작성한 시나리오(화면 흐름)를 검토해 누락된 케이스, 엣지 케이스, UX 개선 포인트를 찾아내는 QA 문서를 만든다.

---

## 입력 처리

두 가지 입력 형식을 지원한다.

**PNG / 이미지**
사용자가 시나리오 전체 캡처 또는 개별 화면 이미지를 첨부한 경우, 이미지를 시각적으로 분석해 각 화면의 이름·내용·흐름을 파악한다. 해상도가 낮아 텍스트를 읽기 어려우면 파악 가능한 범위 안에서 진행하되, 불분명한 부분은 요약 단계에서 "불명확" 으로 표기하고 사용자에게 확인을 요청한다.

**Figma 링크 (MCP)**
`mcp__Figma__get_metadata` 로 노드 구조를 파악한 뒤, 각 섹션/프레임의 텍스트 노드를 추출해 화면 흐름을 재구성한다. 결과가 너무 커서 잘릴 경우 bash로 파일을 열어 텍스트 노드만 추출해 분석한다.

```bash
# 텍스트 노드 추출 예시
cat <결과파일> | python3 -c "
import json, sys, re
data = json.load(sys.stdin)
text = data[0]['text']
texts = re.findall(r'<text[^>]+name=\"([^\"]+)\"', text)
seen = set()
for t in texts:
    if t not in seen and len(t) > 1:
        seen.add(t)
        print(t)
"
```

## QA 프로세스

### 1단계 — 시나리오 요약 (사람이 검증하는 단계)

분석한 내용을 바탕으로 이해한 시나리오를 먼저 정리한다. 이 단계가 중요한 이유는 이후 QA 내용 전체가 이 이해를 전제로 작성되기 때문이다. 내가 잘못 이해했다면 QA도 틀린 방향으로 가게 된다.

다음 형식으로 작성한다.

```
## 시나리오 요약

**플로우**: 화면1 → 화면2 → 화면3 → ... (전체 흐름을 한 줄로)

**주요 액터**: 이 시나리오의 사용자가 누구인지 (예: 선생님, 학생, 관리자)

**핵심 목적**: 이 플로우가 달성하려는 목표 한 문장

**화면 목록**:
- 화면명: 핵심 기능/내용 한 줄 설명
- ...

> 위 내용이 맞는지 확인해주세요. 틀린 부분이 있으면 알려주시면 QA를 다시 진행할게요.
```

### 2단계 — QA 항목 도출

시나리오 요약이 맞다고 확인되면 QA를 진행한다. 확인 없이 자동으로 넘어가도 되지만, 이해가 크게 잘못됐을 경우를 대비해 확인 멘트를 남긴다.

각 항목은 **제목 + 이유**를 함께 작성한다. 이유 없이 제목만 나열하면 기획자가 우선순위를 판단하기 어렵다.

**누락 시나리오** — 플로우상 존재해야 하는데 화면이 없는 경우
- 성공 경로만 있고 실패/오류 경로가 없는 경우
- 신청/제출 후 결과(승인/거절)에 대한 화면이 없는 경우
- 대기 상태(심사 중, 처리 중)에 대한 화면이 없는 경우
- 진입 조건이 충족되지 않았을 때의 화면이 없는 경우

**엣지 케이스** — 정상 흐름은 있지만 예외 상황 처리가 없는 경우
- 사용자가 중간에 이탈 후 재진입했을 때
- 디바이스 권한(카메라, 마이크 등)을 거부했을 때
- 네트워크 오류, 업로드 실패 등 기술적 오류 발생 시
- 동시성 문제 (같은 자원에 여러 명이 동시에 접근할 때)
- 데이터가 없거나 부족한 상태 (empty state)
- 시간 제한이 있는 경우 초과 시 처리

**UI/UX 개선** — 화면은 있지만 사용자 관점에서 개선이 필요한 경우
- 진행 상태(step indicator)가 없어서 사용자가 얼마나 남았는지 모르는 경우
- 되돌릴 수 없는 액션(제출, 저장, 삭제)에 확인 단계가 없는 경우
- 안내 문구가 실제 값 없이 placeholder로 남아있는 경우 (예: "00분")
- 선택 기준이나 제약 조건이 명시되지 않은 경우
- 실수했을 때 복구 경로가 없는 경우

## 출력 형식

결과는 MD 파일로 저장한다. 파일명은 `시나리오명_QA_YYYYMMDD.md` 형식을 사용한다.

```markdown
# [시나리오명] QA 결과

> 작성일: YYYY-MM-DD

---

## 시나리오 요약

**플로우**: ...
**주요 액터**: ...
**핵심 목적**: ...

**화면 목록**:
- 화면명: 설명
- ...

---

## 누락 시나리오

### [제목]
**이유**: 왜 이 화면이 필요한지, 없으면 어떤 문제가 생기는지

### [제목]
**이유**: ...

---

## 엣지 케이스

### [제목]
**이유**: 어떤 상황에서 발생하는지, 현재 처리가 없을 때 사용자가 겪게 될 문제

### [제목]
**이유**: ...

---

## UI/UX 개선

### [제목]
**이유**: 현재 상태의 문제점과 사용자에게 어떤 불편을 주는지

### [제목]
**이유**: ...
```

항목이 없는 카테고리는 "해당 없음"으로 표기한다. 항목 수에 제한은 없지만, 사소한 것보다 기획에 실질적인 영향을 주는 것을 우선적으로 기재한다.

---

## 예시

**입력**: 배달 앱 주문 플로우 PNG 캡처

**시나리오 요약 출력 예시**:
```
플로우: 음식 선택 → 옵션 선택 → 장바구니 → 배달 주소 입력 → 결제 → 주문 완료
주요 액터: 음식 주문 고객
핵심 목적: 원하는 음식을 선택하고 결제까지 완료하는 흐름
```

**QA 항목 예시**:
```
## 누락 시나리오

### 품절 상태 화면
이유: 주문 완료 화면은 있지만 결제 직전 품절이 됐을 때의 안내 화면이 없다.
결제가 된 뒤 취소 처리를 해야 하는 상황이 생기면 사용자 경험이 크게 나빠진다.

## 엣지 케이스

### 결제 중 앱 이탈 후 재진입
이유: 결제 진행 중 홈 버튼을 눌러 나갔다가 돌아왔을 때 주문이 완료됐는지
아닌지 알 수 없는 상태가 된다. 중복 결제나 미완료 주문이 생길 수 있다.

## UI/UX 개선

### 최종 결제 전 확인 단계 없음
이유: "결제하기" 버튼을 누르면 바로 결제가 진행된다. 금액·주소·메뉴를 한 번에
확인할 수 있는 최종 확인 단계가 없어 실수로 잘못된 주소나 수량으로 주문될 수 있다.
```
