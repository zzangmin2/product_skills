# PM 기능 분석 스킬

사용자가 제공한 기능 설명이나 문서를 바탕으로 아래 4단계를 순서대로 수행한다.
각 단계는 이전 단계의 결과를 전제로 하므로, 항상 1 → 2 → 3 → 4 순서를 유지한다.

---

## 분석 단계

### 1단계. 배경과 맥락

이 기능이 왜 만들어지는지를 설명한다. 단순한 기능 요약이 아니라, 아래 질문에 답하는 형식으로 작성한다.

- 이 서비스/기능이 해결하려는 본질적인 문제는 무엇인가?
- 기존 방식(또는 경쟁 서비스)의 어떤 한계를 극복하려 하는가?
- 이 기능이 없으면 서비스 전체에 어떤 영향이 생기는가?
- 비즈니스적으로 왜 지금 이 기능이 중요한가?

**작성 형식:** 짧은 산문 2~4단락. 번호 목록 없이 서술형으로.

---

### 2단계. 문제와 가설

배경에서 도출되는 구체적인 문제들과, 각 문제에 대한 가설을 짝으로 나열한다.

**문제**는 실제로 발생할 수 있는 리스크나 마찰 지점을 의미한다.
추상적인 "품질 문제"가 아니라 "시범 수업 진입 비용이 높아 지원자가 이탈한다"처럼 구체적으로 서술한다.

**가설**은 해당 문제가 왜 발생하는지 또는 어떻게 해결될 수 있는지에 대한 검증 가능한 추측이다.
"H1. ~하면 ~할 것이다" 형식을 사용한다.

**작성 형식:**

```
**문제 N. [문제 제목]**
[문제 서술]

**H[N].** [가설 서술]
```

---

### 3단계. 루트코즈 후보

2단계의 각 문제에 대해 근본 원인 후보를 나열한다.
표면적 원인이 아니라, 그 원인이 왜 발생하는지를 한 단계 더 파고든 것이어야 한다.

예를 들어 "이탈률이 높다"의 루트코즈는 "진입 비용이 높아서"가 아니라,
"진입 비용 대비 기대 수익이 불투명해서" 또는 "대기 시간이 길어 신뢰를 잃어서"처럼 작성한다.

**작성 형식:**

```
**문제 N. [문제 제목]**
- [루트코즈 후보 1]
- [루트코즈 후보 2]
- [루트코즈 후보 3]
```

문제당 루트코즈 후보는 3~5개를 기준으로 한다.

---

### 4단계. 솔루션과 인수 조건

각 문제에 대한 솔루션과, 그 솔루션이 효과 있다고 판단하는 인수 조건(Acceptance Criteria)을 제시한다.

**솔루션**은 루트코즈를 직접 건드리는 것이어야 한다. 표면적 문제를 땜질하는 것이 아니라 원인을 제거하거나 구조를 바꾸는 방향으로 제안한다.

**인수 조건**은 "이 숫자(또는 상태)가 달성되면 이 솔루션은 성공한 것으로 본다"는 합의된 기준이다.
측정 가능한 지표로 작성한다. (ex. "이탈률 20% 이하", "심사 처리 시간 2일 이내 95%")

**작성 형식:**

```
**문제 N. [문제 제목]**

솔루션
- [솔루션 1]
- [솔루션 2]

인수 조건
- [지표 1]
- [지표 2]
```

## 출력 형식

분석 결과는 마크다운 문서로 출력한다.
사용자가 파일로 저장해달라고 하면 `[기능명]_PM분석.md` 파일로 저장한다.

문서 상단에 아래 메타 정보를 표로 포함한다.

```
| 항목 | 내용 |
|------|------|
| 분석 대상 | [기능명] |
| 작성일 | [오늘 날짜] |
| 기반 문서 | [사용자가 제공한 문서명 또는 "직접 입력"] |
```

---

## 주의사항

- 사용자가 4단계 중 일부만 요청해도 (ex. "루트코즈만 뽑아줘"), 해당 단계만 수행한다.
- 사용자가 제공한 맥락이 부족하면 분석 전에 기능의 목적과 대상 사용자를 먼저 확인한다.
- 분석은 서비스의 실제 맥락에 맞게 구체적으로 작성한다. 일반론적인 내용은 지양한다.
